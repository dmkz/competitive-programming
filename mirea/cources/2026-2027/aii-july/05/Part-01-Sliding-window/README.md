Что делать, если нужно рассмотреть много похожих отрезков массива, а считать ответ для каждого с нуля слишком дорого?

Сначала мы разберём самый простой случай, когда длина отрезка уже фиксирована (иначе, окно заданной длины). Потом посмотрим, как и где можно использовать скользящее окно. А затем перейдём к двум указателям, где длина отрезка заранее не известна и постоянно меняется по ходу работы.

## 1. Что такое скользящее окно

Начнём с самой базовой постановки: нужно обработать все подотрезки длины ровно $k$ (например, найти сумму или минимум).

Наивная идея состоит в том, чтобы для каждого окна заново пройти по всем его элементам. Но тогда мы будем много раз пересчитывать почти одно и то же. Но все эти окна идут подряд:

$$
[0, k - 1], [1, k], [2, k + 1], \, \ldots, \, [n - k, n - 1].
$$

Если смотреть на них не как на независимые задачи, а как на последовательность переходов, то видно главное наблюдение: соседние окна отличаются, но не сильно. Когда окно сдвигается на одну позицию вправо, из него уходит только левый элемент, а справа добавляется только один новый элемент.

Именно из этого наблюдения и рождается техника скользящего окна:

- мы рассматриваем окна в естественном порядке;
- не пересчитываем ответ для каждого окна с нуля;
- поддерживаем информацию о текущем окне и обновляем её при переходе к следующему.

То есть здесь важна не сама геометрия отрезков, а идея переиспользования работы, уже сделанной на предыдущем шаге.

Чаще всего скользящее окно используют именно тогда, когда длина окна заранее фиксирована и равна $k$.

После этого всё сводится к очень практическому вопросу: умеем ли мы быстро пересчитывать ответ при одном маленьком изменении окна? Если мы умеем за $O(1)$ или $O(\log(k))$:

- добавить новый элемент справа;
- удалить старый элемент слева;
- получить нужную характеристику окна,

то все окна длины $k$ можно обработать за $O(n)$ или $O(n \log(k))$.

## 2. Самый простой вариант: сумма в каждом окне длины $k$

Лучше всего увидеть технику на характеристике, которая обновляется совсем прозрачно. Самый удобный пример здесь — сумма.

Пусть

$$
s_i = a_i + a_{i+1} + \ldots + a_{i+k-1}
$$

— сумма окна, начинающегося в позиции $i$. Тогда

$$
s_{i+1} = s_i - a_i + a_{i+k}.
$$

Весь смысл скользящего окна в одном равенстве: следующий ответ получается из предыдущего, а не пересчитывается заново.

Значит, при сдвиге окна на один шаг нужно:

- вычесть элемент, который вышел слева;
- прибавить элемент, который вошёл справа.

Маленький рабочий фрагмент на C++:

```cpp
long long window_sum = 0;
// Сначала считаем сумму первого окна.
for (int i = 0; i < k; i++) {
    window_sum += a[i];
}

for (int left = 0; left + k <= n; left++) {
    cout << window_sum << '\n';
    if (left + k < n) {
        // Убираем ушедший слева элемент и добавляем новый справа.
        window_sum -= a[left];
        window_sum += a[left + k];
    }
}
```

И тот же фрагмент на Python3:

```python
# Сначала считаем сумму первого окна.
window_sum = sum(a[:k])

for left in range(n - k + 1):
    print(window_sum)
    if left + k < n:
    # Убираем ушедший слева элемент и добавляем новый справа.
        window_sum -= a[left]
        window_sum += a[left + k]
```

Полная программа для этого базового сюжета: [C++](examples/example-01.cpp), [Python3](examples/example-01.py).

<details>
<summary><strong>Пример ввода и вывода</strong></summary>

Для ввода

```text
8 3
4 2 7 1 3 6 0 5
```

получим вывод

```text
13 10 11 10 9 11
```

</details>

Здесь сложность равна $O(n)$: каждая сумма после первой пересчитывается за $O(1)$.

## 3. Минимум в каждом окне длины $k$: сначала через мультимножество

На сумме всё было слишком удобно: один элемент вышел, один вошёл, и новая сумма сразу выражается формулой.

Теперь специально возьмём характеристику, на которой эта простая арифметика ломается, — минимум. Если из окна ушёл именно минимальный элемент, то новый минимум нельзя восстановить одной формулой.

Значит, сама идея окна остаётся, но рядом с ней появляется новая задача: нужна структура, которая умеет поддерживать текущее содержимое окна. Поэтому нужен контейнер, который умеет:

- вставить новый элемент;
- удалить старый элемент;
- быстро получить минимум.

В C++ самый прямой учебный вариант — стандартное мультимножество:

- вставка работает за $O(\log(k))$;
- удаление одного вхождения работает за $O(\log(k))$;
- минимум всегда лежит в `*st.begin()`.

В Python3 точного аналога мультимножества в стандартной библиотеке нет. Поэтому здесь удобно взять двоичную кучу и ленивое удаление: старые элементы помечаются как удалённые, а реально выбрасываются из кучи только тогда, когда доходят до вершины.

<details>
<summary><strong>Полная программа на C++: минимум в каждом окне через мультимножество</strong></summary>

```cpp
#include <iostream>
#include <set>
#include <vector>
using namespace std;

int main() {
    int n, k;
    cin >> n >> k;
    vector<int> a(n);
    for (int i = 0; i < n; i++) {
        cin >> a[i];
    }

    multiset<int> st;
    // Загружаем в мультимножество первое окно целиком.
    for (int i = 0; i < k; i++) {
        st.insert(a[i]);
    }

    for (int left = 0; left + k <= n; left++) {
        cout << *st.begin();
        if (left + k <= n - 1) {
            cout << ' ';
        }

        if (left + k < n) {
            // Удаляем элемент, который вышел из окна, и добавляем новый.
            st.erase(st.find(a[left]));
            st.insert(a[left + k]);
        }
    }
    cout << '\n';
}
```

</details>

<details>
<summary><strong>Полная программа на Python3: минимум в каждом окне через кучу и ленивое удаление</strong></summary>

```python
import heapq
from collections import defaultdict


n, k = map(int, input().split())
a = list(map(int, input().split()))

heap = []
deleted = defaultdict(int)

# Кладём в кучу первое окно.
for i in range(k):
    heapq.heappush(heap, a[i])

answer = []
for left in range(n - k + 1):
    # Убираем с вершины элементы, которые уже помечены как удалённые.
    while heap and deleted[heap[0]] > 0:
        value = heap[0]
        deleted[value] -= 1
        if deleted[value] == 0:
            del deleted[value]
        heapq.heappop(heap)

    answer.append(heap[0])

    if left + k < n:
        # Левый элемент покидает окно, а новый справа в него входит.
        deleted[a[left]] += 1
        heapq.heappush(heap, a[left + k])

print(*answer)
```

</details>

Файлы примера: [C++](examples/example-02.cpp), [Python3](examples/example-02.py).

Такой подход уже работает и хорошо показывает саму механику скользящего окна фиксированной длины. Но его сложность равна $O(n \log(k))$, а для минимума и максимума можно лучше. Значит, следующий вопрос естественный: нельзя ли хранить в окне не всё подряд, а только действительно полезные элементы.

## 4. Минимум в каждом окне длины $k$: через монотонную двустороннюю очередь

Именно здесь появляется ключевая оптимизация. Для минимума и максимума можно сделать лучше: хранить не все элементы окна, а только тех кандидатов, которые ещё могут стать ответом.

Эта идея уже подробно разбиралась во второй лекции в разделе [Очередь на минимум](../../02/Part-03-Min-queue/README.md#15-очередь-на-минимум). Здесь нам важен именно перенос этой структуры в фиксированное окно длины $k$.

Почему это вообще законно? Если справа пришёл элемент, который не больше какого-то старого элемента, то старый элемент уже никогда не станет минимумом ни в текущем окне, ни в следующих окнах, пока новый элемент находится правее него.

Поэтому будем хранить в двусторонней очереди индексы элементов окна так, чтобы их значения шли по неубыванию.

Когда приходит новый элемент $a_r$:

- пока в конце очереди лежат элементы не меньше $a_r$, они больше не нужны;
- удаляем их;
- добавляем индекс $r$ в конец.

Когда левая граница окна стала равна $l$:

- если в начале очереди лежит индекс меньше $l$, он уже вышел из окна и его нужно удалить.

После этого минимум окна всегда лежит по индексу из начала очереди.

<details>
<summary><strong>Полная программа на C++: минимум в каждом окне через двустороннюю очередь</strong></summary>

```cpp
#include <deque>
#include <iostream>
#include <vector>
using namespace std;

int main() {
    int n, k;
    cin >> n >> k;
    vector<int> a(n);
    for (int i = 0; i < n; i++) {
        cin >> a[i];
    }

    deque<int> dq;
    // В двусторонней очереди храним только кандидатов на минимум.
    for (int i = 0; i < n; i++) {
        // Выкидываем из конца все элементы не лучше текущего.
        while (!dq.empty() && a[dq.back()] >= a[i]) {
            dq.pop_back();
        }
        dq.push_back(i);

        int left = i - k + 1;
        if (left < 0) {
            continue;
        }
        // Убираем из начала индексы, которые уже вышли из окна.
        while (!dq.empty() && dq.front() < left) {
            dq.pop_front();
        }

        cout << a[dq.front()];
        if (i + 1 < n) {
            if (left + 1 <= n - k) {
                cout << ' ';
            }
        }
    }
    cout << '\n';
}
```

</details>

<details>
<summary><strong>Полная программа на Python3: минимум в каждом окне через двустороннюю очередь</strong></summary>

```python
from collections import deque


n, k = map(int, input().split())
a = list(map(int, input().split()))

dq = deque()
answer = []

for i in range(n):
    # Выкидываем из конца все элементы не лучше текущего.
    while dq and a[dq[-1]] >= a[i]:
        dq.pop()
    dq.append(i)

    left = i - k + 1
    if left < 0:
        continue

    # Убираем из начала индексы, которые уже вышли из окна.
    while dq and dq[0] < left:
        dq.popleft()
    answer.append(a[dq[0]])

print(*answer)
```

</details>

Файлы примера: [C++](examples/example-03.cpp), [Python3](examples/example-03.py).

Здесь сложность уже равна $O(n)$, потому что каждый индекс добавляется в двустороннюю очередь и удаляется из неё не более одного раза. Для окна фиксированной длины это уже оптимальный по порядку вариант.

## 5. Что ещё умеет фиксированное окно, а чего не умеет само по себе

Перед тем как идти дальше, полезно зафиксировать не только силу метода, но и его границы.

Скользящее окно длины $k$ полезно тогда, когда ответ по окну можно поддерживать локально. Типичные примеры:

- сумма элементов окна;
- количество нулей, положительных чисел или вообще элементов с каким-то признаком;
- количество различных значений, если поддерживать словарь частот;
- минимум или максимум, если взять мультимножество или монотонную двустороннюю очередь;
- несколько характеристик сразу, если каждая из них умеет обновляться при добавлении и удалении элемента.

> [!IMPORTANT]
> Но само по себе скользящее окно не является магией. Оно не отвечает на вопрос, как быстро удалять левый элемент для произвольной функции от окна.

Например:

- для минимума и максимума понадобилась отдельная идея с кандидатами;
- для количества различных значений нужен словарь частот;
- для медианы или $k$-й порядковой статистики уже нужны более тяжёлые структуры.

> [!NOTE]
> Ещё одно важное ограничение: в этой части длина окна была фиксирована заранее. Как только в задаче нужно не обработать данное $k$, а как-то выбирать длину по ходу, картина меняется. Следующий шаг как раз про это: сначала посмотрим, как окно превращается в быструю проверку внутри бинарного поиска по ответу.
