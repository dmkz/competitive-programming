# Лекция 5. Два указателя и скользящее окно

Что делать, если нужно рассмотреть много похожих отрезков массива, а считать ответ для каждого с нуля слишком дорого?

Сначала мы разберём самый простой случай, когда длина отрезка уже фиксирована (иначе, окно заданной длины). Потом посмотрим, как и где можно использовать скользящее окно. А затем перейдём к двум указателям, где длина отрезка заранее не известна и постоянно меняется по ходу работы.

## 1. Что такое скользящее окно

Начнём с самой базовой постановки: нужно обработать все подотрезки длины ровно $k$ (например, найти сумму или минимум).

Наивная идея состоит в том, чтобы для каждого окна заново пройти по всем его элементам. Но тогда мы будем много раз пересчитывать почти одно и то же. Но все эти окна идут подряд:

$$
[0, k - 1], [1, k], [2, k + 1], \, \ldots, \, [n - k, n - 1].
$$

Если смотреть на них не как на независимые задачи, а как на последовательность переходов, то видно главное наблюдение: соседние окна отличаются, но не сильно. Когда окно сдвигается на одну позицию вправо, из него уходит только левый элемент, а справа добавляется только один новый элемент.

Именно из этого наблюдения и рождается техника скользящего окна:

- мы рассматриваем окна в естественном порядке;
- не пересчитываем ответ для каждого окна с нуля;
- поддерживаем информацию о текущем окне и обновляем её при переходе к следующему.

То есть здесь важна не сама геометрия отрезков, а идея переиспользования работы, уже сделанной на предыдущем шаге.

Чаще всего скользящее окно используют именно тогда, когда длина окна заранее фиксирована и равна $k$.

После этого всё сводится к очень практическому вопросу: умеем ли мы быстро пересчитывать ответ при одном маленьком изменении окна? Если мы умеем за $O(1)$ или $O(\log(k))$:

- добавить новый элемент справа;
- удалить старый элемент слева;
- получить нужную характеристику окна,

то все окна длины $k$ можно обработать за $O(n)$ или $O(n \log(k))$.

## 2. Самый простой вариант: сумма в каждом окне длины $k$

Лучше всего увидеть технику на характеристике, которая обновляется совсем прозрачно. Самый удобный пример здесь — сумма.

Пусть

$$
s_i = a_i + a_{i+1} + \ldots + a_{i+k-1}
$$

— сумма окна, начинающегося в позиции $i$. Тогда

$$
s_{i+1} = s_i - a_i + a_{i+k}.
$$

Весь смысл скользящего окна в одном равенстве: следующий ответ получается из предыдущего, а не пересчитывается заново.

Значит, при сдвиге окна на один шаг нужно:

- вычесть элемент, который вышел слева;
- прибавить элемент, который вошёл справа.

Маленький рабочий фрагмент на C++:

```cpp
long long window_sum = 0;
// Сначала считаем сумму первого окна.
for (int i = 0; i < k; i++) {
    window_sum += a[i];
}

for (int left = 0; left + k <= n; left++) {
    cout << window_sum << '\n';
    if (left + k < n) {
        // Убираем ушедший слева элемент и добавляем новый справа.
        window_sum -= a[left];
        window_sum += a[left + k];
    }
}
```

И тот же фрагмент на Python3:

```python
# Сначала считаем сумму первого окна.
window_sum = sum(a[:k])

for left in range(n - k + 1):
    print(window_sum)
    if left + k < n:
    # Убираем ушедший слева элемент и добавляем новый справа.
        window_sum -= a[left]
        window_sum += a[left + k]
```

Полная программа для этого базового сюжета: [C++](Part-01-Sliding-window/examples/example-01.cpp), [Python3](Part-01-Sliding-window/examples/example-01.py).

## 3. Минимум в каждом окне длины $k$: сначала через мультимножество

На сумме всё было слишком удобно: один элемент вышел, один вошёл, и новая сумма сразу выражается формулой.

Теперь специально возьмём характеристику, на которой эта простая арифметика ломается, — минимум. Если из окна ушёл именно минимальный элемент, то новый минимум нельзя восстановить одной формулой.

Значит, сама идея окна остаётся, но рядом с ней появляется новая задача: нужна структура, которая умеет поддерживать текущее содержимое окна. Поэтому нужен контейнер, который умеет:

- вставить новый элемент;
- удалить старый элемент;
- быстро получить минимум.

В C++ самый прямой учебный вариант — стандартное мультимножество:

- вставка работает за $O(\log(k))$;
- удаление одного вхождения работает за $O(\log(k))$;
- минимум всегда лежит в `*st.begin()`.

В Python3 точного аналога мультимножества в стандартной библиотеке нет. Поэтому здесь удобно взять двоичную кучу и ленивое удаление: старые элементы помечаются как удалённые, а реально выбрасываются из кучи только тогда, когда доходят до вершины.

<details>
<summary><strong>Полная программа на C++: минимум в каждом окне через мультимножество</strong></summary>

```cpp
#include <iostream>
#include <set>
#include <vector>
using namespace std;

int main() {
    int n, k;
    cin >> n >> k;
    vector<int> a(n);
    for (int i = 0; i < n; i++) {
        cin >> a[i];
    }

    multiset<int> st;
    // Загружаем в мультимножество первое окно целиком.
    for (int i = 0; i < k; i++) {
        st.insert(a[i]);
    }

    for (int left = 0; left + k <= n; left++) {
        cout << *st.begin();
        if (left + k <= n - 1) {
            cout << ' ';
        }

        if (left + k < n) {
            // Удаляем элемент, который вышел из окна, и добавляем новый.
            st.erase(st.find(a[left]));
            st.insert(a[left + k]);
        }
    }
    cout << '\n';
}
```

</details>

<details>
<summary><strong>Полная программа на Python3: минимум в каждом окне через кучу и ленивое удаление</strong></summary>

```python
import heapq
from collections import defaultdict


n, k = map(int, input().split())
a = list(map(int, input().split()))

heap = []
deleted = defaultdict(int)

# Кладём в кучу первое окно.
for i in range(k):
    heapq.heappush(heap, a[i])

answer = []
for left in range(n - k + 1):
    # Убираем с вершины элементы, которые уже помечены как удалённые.
    while heap and deleted[heap[0]] > 0:
        value = heap[0]
        deleted[value] -= 1
        if deleted[value] == 0:
            del deleted[value]
        heapq.heappop(heap)

    answer.append(heap[0])

    if left + k < n:
        # Левый элемент покидает окно, а новый справа в него входит.
        deleted[a[left]] += 1
        heapq.heappush(heap, a[left + k])

print(*answer)
```

</details>

Файлы примера: [C++](Part-01-Sliding-window/examples/example-02.cpp), [Python3](Part-01-Sliding-window/examples/example-02.py).

Такой подход уже работает и хорошо показывает саму механику скользящего окна фиксированной длины. Но его сложность равна $O(n \log(k))$, а для минимума и максимума можно лучше. Значит, следующий вопрос естественный: нельзя ли хранить в окне не всё подряд, а только действительно полезные элементы.

## 4. Минимум в каждом окне длины $k$: через монотонную двустороннюю очередь

Именно здесь появляется ключевая оптимизация. Для минимума и максимума можно сделать лучше: хранить не все элементы окна, а только тех кандидатов, которые ещё могут стать ответом.

Эта идея уже подробно разбиралась во второй лекции в разделе [Очередь на минимум](Part-01-Sliding-window/../../02/Part-03-Min-queue/README.md#15-очередь-на-минимум). Здесь нам важен именно перенос этой структуры в фиксированное окно длины $k$.

Почему это вообще законно? Если справа пришёл элемент, который не больше какого-то старого элемента, то старый элемент уже никогда не станет минимумом ни в текущем окне, ни в следующих окнах, пока новый элемент находится правее него.

Поэтому будем хранить в двусторонней очереди индексы элементов окна так, чтобы их значения шли по неубыванию.

Когда приходит новый элемент $a_r$:

- пока в конце очереди лежат элементы не меньше $a_r$, они больше не нужны;
- удаляем их;
- добавляем индекс $r$ в конец.

Когда левая граница окна стала равна $l$:

- если в начале очереди лежит индекс меньше $l$, он уже вышел из окна и его нужно удалить.

После этого минимум окна всегда лежит по индексу из начала очереди.

<details>
<summary><strong>Полная программа на C++: минимум в каждом окне через двустороннюю очередь</strong></summary>

```cpp
#include <deque>
#include <iostream>
#include <vector>
using namespace std;

int main() {
    int n, k;
    cin >> n >> k;
    vector<int> a(n);
    for (int i = 0; i < n; i++) {
        cin >> a[i];
    }

    deque<int> dq;
    // В двусторонней очереди храним только кандидатов на минимум.
    for (int i = 0; i < n; i++) {
        // Выкидываем из конца все элементы не лучше текущего.
        while (!dq.empty() && a[dq.back()] >= a[i]) {
            dq.pop_back();
        }
        dq.push_back(i);

        int left = i - k + 1;
        if (left < 0) {
            continue;
        }
        // Убираем из начала индексы, которые уже вышли из окна.
        while (!dq.empty() && dq.front() < left) {
            dq.pop_front();
        }

        cout << a[dq.front()];
        if (i + 1 < n) {
            if (left + 1 <= n - k) {
                cout << ' ';
            }
        }
    }
    cout << '\n';
}
```

</details>

<details>
<summary><strong>Полная программа на Python3: минимум в каждом окне через двустороннюю очередь</strong></summary>

```python
from collections import deque


n, k = map(int, input().split())
a = list(map(int, input().split()))

dq = deque()
answer = []

for i in range(n):
    # Выкидываем из конца все элементы не лучше текущего.
    while dq and a[dq[-1]] >= a[i]:
        dq.pop()
    dq.append(i)

    left = i - k + 1
    if left < 0:
        continue

    # Убираем из начала индексы, которые уже вышли из окна.
    while dq and dq[0] < left:
        dq.popleft()
    answer.append(a[dq[0]])

print(*answer)
```

</details>

Файлы примера: [C++](Part-01-Sliding-window/examples/example-03.cpp), [Python3](Part-01-Sliding-window/examples/example-03.py).

Здесь сложность уже равна $O(n)$, потому что каждый индекс добавляется в двустороннюю очередь и удаляется из неё не более одного раза. Для окна фиксированной длины это уже оптимальный по порядку вариант.

## 5. Что ещё умеет фиксированное окно, а чего не умеет само по себе

Перед тем как идти дальше, полезно зафиксировать не только силу метода, но и его границы.

Скользящее окно длины $k$ полезно тогда, когда ответ по окну можно поддерживать локально. Типичные примеры:

- сумма элементов окна;
- количество нулей, положительных чисел или вообще элементов с каким-то признаком;
- количество различных значений, если поддерживать словарь частот;
- минимум или максимум, если взять мультимножество или монотонную двустороннюю очередь;
- несколько характеристик сразу, если каждая из них умеет обновляться при добавлении и удалении элемента.

> [!IMPORTANT]
> Но само по себе скользящее окно не является магией. Оно не отвечает на вопрос, как быстро удалять левый элемент для произвольной функции от окна.

Например:

- для минимума и максимума понадобилась отдельная идея с кандидатами;
- для количества различных значений нужен словарь частот;
- для медианы или $k$-й порядковой статистики уже нужны более тяжёлые структуры.

> [!NOTE]
> Ещё одно важное ограничение: в этой части длина окна была фиксирована заранее. Как только в задаче нужно не обработать данное $k$, а как-то выбирать длину по ходу, картина меняется. Следующий шаг как раз про это: сначала посмотрим, как окно превращается в быструю проверку внутри бинарного поиска по ответу.

## 6. Скользящее окно внутри бинарного поиска

В предыдущей части длина окна $k$ была уже дана. Теперь перейдём к следующему типу задач: длину нужно не использовать, а найти.

Очень часто (почти всегда) задача спрашивает не ответ для одного фиксированного $k$, а **максимальное** или **минимальное** значение длины.

Здесь одного скользящего окна уже недостаточно. Оно хорошо отвечает на вопрос “что происходит при фиксированном $k$?”, но не выбирает $k$ само. Поэтому типичный маршрут рассуждения выглядит так:

- существует ли подходящий подотрезок длины ровно $k$;
- найдите максимальное $k$, для которого такой подотрезок существует.

В таких случаях сперва проверяют монотонность: если подходящий подотрезок длины $k$ существует, то подходящий подотрезок длины $k - 1$ тоже существует? А если подходящего подотрезка длины $k$ не существует, то подходящего подотрезка длины $k + 1$ уж точно не существует, да? Если да, тогда появляется стандартная схема:

1. бинарным поиском перебираем длину $k$;
2. для фиксированного $k$ запускаем быструю проверку;
3. внутри этой проверки часто и живёт скользящее окно длины ровно $k$.

То есть роли здесь строго разделены: скользящее окно отвечает за быстрый проход по всем окнам фиксированной длины, а бинарный поиск отвечает за переход от задачи проверки к задаче оптимизации.

---

## 7. Пример: максимальная длина подотрезка положительных чисел с суммой не больше $x$

Разберём эту схему на задаче, где хорошо видно и проверку, и монотонность. Пусть все элементы массива положительны, и нужно найти максимальную длину подотрезка, сумма которого не превосходит $x$.

Если бы длина $k$ была дана заранее, мы бы спрашивали только одно:

> существует ли подотрезок длины ровно $k$ с суммой не больше $x$?

И вот здесь снова появляется уже знакомое окно фиксированной длины. Этот вопрос можно проверить за $O(n)$:

- сначала считаем сумму первого окна;
- затем сдвигаем окно на один шаг;
- если хотя бы у одного окна сумма не больше $x$, то проверка успешна.

Теперь главный вопрос не про реализацию проверки, а про то, почему вообще здесь разрешено искать ответ бинарным поиском по $k$.

Потому что если существует подходящий подотрезок длины $k$, то у него есть и любой префикс длины меньше $k$, а сумма этого префикса тоже не превосходит $x$, так как все элементы положительны.

Значит:

- если длина $k$ подходит, то все меньшие длины тоже подходят;
- если длина $k$ не подходит, то все большие длины тоже не подходят.

Именно это и даёт монотонность для бинарного поиска.

Маленький фрагмент функции проверки на C++:

```cpp
bool check(const vector<int>& a, int k, long long x) {
    // Считаем сумму первого окна длины k.
    long long window_sum = 0;
    for (int i = 0; i < k; i++) {
        window_sum += a[i];
    }
    if (window_sum <= x) {
        return true;
    }

    for (int left = 0; left + k < (int)a.size(); left++) {
        // Переходим к следующему окну той же длины.
        window_sum -= a[left];
        window_sum += a[left + k];
        if (window_sum <= x) {
            return true;
        }
    }
    return false;
}
```

Тот же фрагмент на Python3:

```python
def check(a, k, x):
    # Считаем сумму первого окна длины k.
    window_sum = sum(a[:k])
    if window_sum <= x:
        return True

    for left in range(len(a) - k):
        # Переходим к следующему окну той же длины.
        window_sum -= a[left]
        window_sum += a[left + k]
        if window_sum <= x:
            return True
    return False
```

```cpp
using namespace std;

bool check(const vector<int>& a, int k, long long x) {

        return true;

    int high = n + 1;
    // На полуинтервале [low, high): low подходит, high ещё не подходит.
    while (high - low > 1) {

        if (check(a, mid, x)) {
            low = mid;
        } else {
            high = mid;
        }
    }
```

```python
low = 0
high = n + 1
# На полуинтервале [low, high): low подходит, high ещё не подходит.
while high - low > 1:
    mid = (low + high) // 2
    if check(a, mid, x):
        low = mid
    else:
        high = mid

print(low)
```

</details>

Такое решение работает за $O(n \log(n))$:

- одна проверка занимает $O(n)$;
- бинарный поиск делает $O(\log(n))$ проверок.

## 8. Когда эта схема полезна, а когда даёт лишний логарифм

> [!NOTE]
> На этом примере важно увидеть не только сам шаблон, но и его цену.

Схема “бинарный поиск по ответу + скользящее окно” полезна, когда:

- есть чёткая монотонность по длине;
- для фиксированной длины $k$ мы умеем быстро проверить все окна;
- прямого линейного решения не видно или оно сложнее в реализации.

Внутри такой проверки можно использовать не только сумму окна. Подойдут и другие характеристики:

- минимум или максимум в окне;
- количество различных значений;
- число элементов с нужным признаком;
- какая-то комбинация нескольких локально поддерживаемых величин.

> [!WARNING]
> Если прямое решение за $O(n)$ уже очевидно и вы его знаете, то по асимптотике оно лучше, чем решение за $O(n \log(n))$.
>
> Но на олимпиадах важна не только асимптотика. Решение через бинарный поиск по ответу часто проще придумать и быстрее написать: внешний цикл стандартный, а проверка для фиксированного $k$ нередко заметно проще исходной задачи.
>
> Поэтому в соревновании не нужно любой ценой дожимать линейное решение, если вариант с бинарным поиском уже надёжен, понятен и уверенно проходит в лимит.
>
> Когда ограничение по времени равно `2 секундам`, разница между `30 мс` и `300 мс` обычно не играет роли. Гораздо важнее быстро и без ошибок сдать эту задачу и перейти к следующим.

В образовательных целях эту же задачу полезно решить ещё раз другим способом. В следующей части мы уберём бинарный поиск совсем и разрешим длине окна меняться по ходу работы.

## 9. Что такое метод двух указателей

В прошлой части мы специально оставили открытым вопрос: можно ли убрать лишний логарифм и не перебирать длину бинарным поиском.

Можно, если перестать фиксировать длину заранее и разрешить отрезку сжиматься и расширяться по ходу алгоритма. Именно это и делает метод двух указателей.

Идея метода двух указателей:

- держим текущий отрезок $[l, r]$;
- умеем быстро расширять его вправо;
- умеем быстро сдвигать его левую границу;
- следим, чтобы оба указателя в сумме сделали не больше $O(n)$ шагов.

Главное отличие от фиксированного окна:

- в скользящем окне длина заранее зафиксирована;
- в методе двух указателей длина выбирается по ходу работы алгоритма.

То есть теперь мы уже не перебираем все окна одной длины. Мы ведём один живой отрезок и всё время пытаемся держать его в корректном состоянии.

Очень часто один указатель отвечает за “попробуем добавить ещё один элемент справа”, а второй — за “нужно убрать что-то слева, чтобы условие снова стало верным”.

---

## 10. Когда два указателя на одном массиве работают хорошо

Но здесь легко сделать слишком резкий переход и начать применять два указателя везде подряд. Так делать нельзя. Сначала надо понять, откуда вообще берётся линейность.

Наиболее типичная ситуация такая:

- массив обрабатывается слева направо;
- правая граница только увеличивается;
- левая граница тоже только увеличивается;
- после каждого движения можно локально обновить информацию о текущем окне.

Чаще всего такой подход работает в задачах, где нужно:

- найти максимальную длину подотрезка, у которого нужная величина не превосходит заданный порог;
- найти минимальную длину подотрезка, у которого нужная величина уже не меньше заданного порога;
- посчитать количество подходящих подотрезков;
- поддерживать в окне сумму, число различных значений, число плохих элементов и другие величины.

> [!IMPORTANT]
> За этим списком стоит важное скрытое условие: после сдвига левой границы ситуация должна становиться не хуже в том смысле, который нужен задаче. Иначе указатели не будут двигаться монотонно, а вся линейность развалится.

## 11. Пример: максимальная длина подотрезка положительных чисел с суммой не больше $x$

Вернёмся к той же задаче, что и в предыдущей части. Это полезно сделать специально: так лучше видно, как схема “бинарный поиск + проверка” схлопывается в один проход.

Итак:

- дан массив положительных чисел;
- нужно найти максимальную длину подотрезка с суммой не больше $x$.

В части про бинарный поиск мы решали её за $O(n \log(n))$. Но для положительных чисел здесь есть более сильная идея.

Пусть текущая сумма окна равна $s$.

- Двигаем правую границу вправо и добавляем новый элемент.
- Пока $s > x$, двигаем левую границу вправо и вычитаем элементы.
- Как только снова получилось $s \le x$, текущее окно допустимо, и можно обновить ответ.

Это работает именно потому, что все элементы положительны:

- при расширении окна сумма только растёт;
- при сдвиге левой границы сумма только уменьшается.

Маленький рабочий фрагмент на C++:

```cpp
int left = 0;
long long window_sum = 0;
int answer = 0;

for (int right = 0; right < n; right++) {
    // Расширяем окно вправо.
    window_sum += a[right];
    // Пока окно плохое, сдвигаем левую границу.
    while (window_sum > x) {
        window_sum -= a[left];
        left++;
    }
    answer = max(answer, right - left + 1);
}
```

И тот же фрагмент на Python3:

```python
left = 0
window_sum = 0
answer = 0

for right in range(n):
    # Расширяем окно вправо.
    window_sum += a[right]
    # Пока окно плохое, сдвигаем левую границу.
    while window_sum > x:
        window_sum -= a[left]
        left += 1
    answer = max(answer, right - left + 1)
```

## 12. Ещё один пример: удаление повторов в отсортированном массиве

> [!NOTE]
> Важно не привязывать два указателя только к историям про окна и суммы. У этой техники есть и другой очень практический вид: один указатель читает данные, а второй перестраивает ответ на месте.

Ещё один классический сюжет — обработка уже отсортированного массива.

Пусть массив отсортирован в порядке неубывания, и нужно перенести все различные значения в его начало, оставив по одному вхождению каждого значения.

Заведём два указателя:

- первый по очереди считывает элементы массива;
- второй показывает, куда записывать следующее новое значение.

Если очередной считанный элемент отличается от последнего уже записанного значения, то:

- записываем его в следующую свободную позицию;
- сдвигаем указатель записи.

В конце начальная часть массива будет содержать все различные значения по одному разу. После этого обычно удобно явно сжать массив до длины `write`, чтобы дальше работать уже с готовым ответом.

<details>
<summary><strong>Полная программа на C++</strong></summary>

```cpp
#include <iostream>
#include <vector>
using namespace std;

int main() {
    int n;
    cin >> n;
    vector<int> a(n);
    for (int i = 0; i < n; i++) {
        cin >> a[i];
    }

    int write = 0;
    // Один указатель читает массив, второй показывает место для следующего нового значения.
    for (int read = 0; read < n; read++) {
        if (read == 0 || a[read] != a[write - 1]) {
            // Переносим очередное новое значение в начало массива.
            a[write] = a[read];
            write++;
        }
    }

    a.resize(write);

    cout << a.size() << '\n';
    for (int i = 0; i < (int)a.size(); i++) {
        if (i > 0) {
            cout << ' ';
        }
        cout << a[i];
    }
    cout << '\n';
}
```

</details>

<summary><strong>Полная программа на Python3</strong></summary>

## 13. Что ещё можно делать двумя указателями на одном массиве

Теперь коротко перечислим, где два указателя на одном массиве встречаются особенно часто. Вот несколько типичных сюжетов:

- поиск максимальной длины подотрезка, у которого нужная величина не превосходит заданный порог;
- поиск минимальной длины подотрезка, у которого нужная величина уже не меньше заданного порога;
- подсчёт количества подходящих подотрезков;
- поддержание в окне суммы, числа различных значений, числа плохих элементов, максимума, минимума и других величин;
- сжатие отсортированных данных, объединение равных блоков, перенос нужных элементов в начало.

> [!WARNING]
> Но важно помнить и о границах применимости:
>
> - если условие не становится проще после сдвига левой границы, метод может не работать;
> - если в задаче на суммы есть отрицательные числа, стандартная монотонность часто ломается;
> - если правую границу приходится часто двигать назад, линейности уже не будет.

Но этим возможности метода не исчерпываются. Есть ещё важный класс задач после сортировки, где два указателя тоже работают хорошо, хотя привычного окна там уже нет. К этим задачам мы сейчас и перейдём.

## 14. Два указателя после сортировки

До сих пор оба указателя двигались по одному и тому же массиву слева направо и обслуживали текущий отрезок. После сортировки картина меняется.

Теперь два указателя могут вообще не описывать окно. Их задача другая: смотреть на упорядоченные данные так, чтобы каждым движением отбрасывать сразу целую группу бесполезных вариантов.

После сортировки появляются ещё две очень частые схемы:

- один указатель идёт с начала массива, а второй — с конца;
- по одному указателю идёт в каждом из двух отсортированных массивов.

В этих задачах уже нет “окна” в привычном смысле. Указатели нужны не для поддержки текущего подотрезка, а для того, чтобы отбрасывать заведомо бесполезные варианты целыми группами.

## 15. Пример: существует ли пара с суммой $x$

Самый классический пример такого типа — поиск пары с заданной суммой.

Пусть дан массив целых чисел и число $x$. Нужно понять, существует ли пара элементов с суммой ровно $x$.

После сортировки массива берём два указателя:

- `left` стоит в начале;
- `right` стоит в конце.

Если

$$
a_{left} + a_{right} < x,
$$

то левый элемент слишком мал, и его нужно увеличить, то есть сдвинуть `left` вправо.

Если

$$
a_{left} + a_{right} > x,
$$

то правый элемент слишком велик, и его нужно уменьшить, то есть сдвинуть `right` влево.

Если же сумма равна $x$, ответ найден.

<details>
<summary><strong>Полная программа на C++</strong></summary>

```cpp
#include <algorithm>
#include <iostream>
#include <vector>
using namespace std;

int main() {
    int n;
    long long x;
    cin >> n >> x;
    vector<long long> a(n);
    for (int i = 0; i < n; i++) {
        cin >> a[i];
    }

    sort(a.begin(), a.end());

    int left = 0;
    int right = n - 1;
    bool ok = false;
    while (left < right) {
        long long sum = a[left] + a[right];
        if (sum == x) {
            ok = true;
            break;
        }
        if (sum < x) {
            // Левый элемент слишком мал, увеличиваем сумму.
            left++;
        } else {
            // Правый элемент слишком велик, уменьшаем сумму.
            right--;
        }
    }

    cout << (ok ? "YES" : "NO") << '\n';
}
```

</details>

<details>
<summary><strong>Полная программа на Python3</strong></summary>


Если сортировка уже дана, то сам проход двумя указателями занимает $O(n)$. Если сортировку нужно сделать самим, то полная сложность равна $O(n \log(n))$.

## 16. Пример: пересечение двух массивов

Следующий стандартный сюжет — два указателя уже не на одном массиве, а на двух отсортированных массивах сразу.

Пусть даны два массива. Нужно вывести все общие элементы с учётом кратности.

После сортировки обоих массивов берём два указателя:

- `i` идёт по первому массиву;
- `j` идёт по второму массиву.

Если:

- $a_i = b_j$, то этот элемент входит в пересечение, и оба указателя сдвигаются;
- $a_i < b_j$, то нужно двигать `i`;
- $a_i > b_j$, то нужно двигать `j`.

> [!TIP]
> В C++ этот же шаблон уже есть в стандартной библиотеке: это `std::set_intersection()` для двух отсортированных диапазонов.

<details>
<summary><strong>Полная программа на C++</strong></summary>

```cpp
#include <algorithm>
#include <iostream>
#include <vector>
using namespace std;

int main() {
    int n, m;
    cin >> n >> m;
    vector<int> a(n), b(m);
    for (int i = 0; i < n; i++) {
        cin >> a[i];
    }
    for (int i = 0; i < m; i++) {
        cin >> b[i];
    }

    sort(a.begin(), a.end());
    sort(b.begin(), b.end());

    vector<int> common;
    int i = 0;
    int j = 0;
    // Сравниваем текущие элементы и двигаем тот указатель, где значение меньше.
    while (i < n && j < m) {
        if (a[i] == b[j]) {
            common.push_back(a[i]);
            i++;
            j++;
        } else if (a[i] < b[j]) {
            i++;
        } else {
            j++;
        }
    }

    cout << common.size() << '\n';
    for (int k = 0; k < (int)common.size(); k++) {
        if (k > 0) {
            cout << ' ';
        }
        cout << common[k];
    }
    cout << '\n';
}
```

</details>

<details>
<summary><strong>Полная программа на Python3</strong></summary>

## 17. Пример: слияние двух отсортированных массивов

Есть ещё один очень частый сюжет с двумя указателями на двух массивах: нужно не найти общие элементы, а слить оба отсортированных массива в один общий отсортированный массив.

Пусть даны два массива, каждый уже отсортирован в порядке неубывания.

Заведём два указателя:

- `i` идёт по первому массиву;
- `j` идёт по второму массиву.

Пока оба указателя стоят на допустимых позициях, сравниваем текущие элементы:

- если $a_i \le b_j$, берём элемент из первого массива и двигаем `i`;
- иначе берём элемент из второго массива и двигаем `j`.

Когда один из массивов закончился, остаётся просто дописать хвост второго.

Это и есть операция `merge` из сортировки слиянием. В C++ для неё есть и готовая функция `std::merge`, но руками этот шаблон тоже надо уметь писать.

<details>
<summary><strong>Полная программа на C++</strong></summary>

```cpp
#include <iostream>
#include <vector>
using namespace std;

int main() {
    int n, m;
    cin >> n >> m;
    vector<int> a(n), b(m);
    for (int i = 0; i < n; i++) {
        cin >> a[i];
    }
    for (int j = 0; j < m; j++) {
        cin >> b[j];
    }

    vector<int> merged;
    merged.reserve(n + m);

    int i = 0;
    int j = 0;
    while (i < n && j < m) {
        if (a[i] <= b[j]) {
            // Берём меньший текущий элемент из первого массива.
            merged.push_back(a[i]);
            i++;
        } else {
            // Берём меньший текущий элемент из второго массива.
            merged.push_back(b[j]);
            j++;
        }
    }

    while (i < n) {
        merged.push_back(a[i]);
        i++;
    }
    while (j < m) {
        merged.push_back(b[j]);
        j++;
    }

    for (int k = 0; k < (int)merged.size(); k++) {
        if (k > 0) {
            cout << ' ';
        }
        cout << merged[k];
    }
    cout << '\n';
}
```

</details>

<summary><strong>Полная программа на Python3</strong></summary>

## 18. Ещё один пример: количество пар с ограниченной суммой

Этот сюжет уже встречался в [задаче E из первого тематического контеста](Part-04-Sorted-two-pointers-and-checklist/../../01/contest/README.md). В самой задаче нужно посчитать количество пар $(i, j)$, где $i < j$ и $a_i + a_j \le s$.

Если нужен строгий вариант $a_i + a_j < s$, то идея остаётся той же самой: меняется только знак в проверке.

В тематическом контесте эта задача решалась через бинарный поиск: сортируем массив, фиксируем правый элемент пары и бинарным поиском считаем, сколько левых элементов подходят на префиксе.

Но после сортировки здесь можно пройти и двумя указателями.

Поставим:

- `left` в начало массива;
- `right` в конец массива.

Дальше смотрим на сумму $a_{left} + a_{right}$.

Если

$$
a_{left} + a_{right} \le s,
$$

то подходит не только пара `(left, right)`, но и вообще все пары `(left, k)`, где `left < k \le right`.

Причина простая: массив отсортирован, значит для любого такого `k` выполнено $a_k \le a_{right}$, а тогда

$$
a_{left} + a_k \le a_{left} + a_{right} \le s.
$$

Значит, можно сразу прибавить к ответу `right - left` пар и сдвинуть `left` вправо.

Если же

$$
a_{left} + a_{right} > s,
$$

то правый элемент слишком велик, и нужно уменьшать сумму, то есть сдвигать `right` влево.

<details>
<summary><strong>Полная программа на C++</strong></summary>

```cpp
#include <algorithm>
#include <iostream>
#include <vector>
using namespace std;

void solve() {
    int n;
    long long s;
    cin >> n >> s;
    vector<long long> a(n);
    for (int i = 0; i < n; i++) {
        cin >> a[i];
    }

    sort(a.begin(), a.end());

    int left = 0;
    int right = n - 1;
    long long answer = 0;
    while (left < right) {
        long long sum = a[left] + a[right];
        if (sum <= s) {
            // Для фиксированного left подходят все вторые индексы от left + 1 до right.
            answer += right - left;
            left++;
        } else {
            // Правый элемент слишком велик, уменьшаем сумму.
            right--;
        }
    }

    cout << answer << '\n';
}

int main() {
    int tests;
    cin >> tests;
    while (tests-- > 0) {
        solve();
    }
}
```

</details>

<details>
<summary><strong>Полная программа на Python3</strong></summary>

```python
def solve():
    n, s = map(int, input().split())
    a = list(map(int, input().split()))

    a.sort()

    left = 0
    right = n - 1
    answer = 0
    while left < right:
        current_sum = a[left] + a[right]
        if current_sum <= s:
            # Для фиксированного left подходят все вторые индексы от left + 1 до right.
            answer += right - left
            left += 1
        else:
            # Правый элемент слишком велик, уменьшаем сумму.
            right -= 1

    print(answer)


tests = int(input())
for _ in range(tests):
    solve()
```

Файлы примера: [C++](Part-04-Sorted-two-pointers-and-checklist/examples/example-03.cpp), [Python3](Part-04-Sorted-two-pointers-and-checklist/examples/example-03.py).

После сортировки проход занимает $O(n)$, поэтому полная сложность решения равна $O(n \log(n))$.

Заметьте, что у решения из контестного разбора общая асимптотика такая же: сортировка всё равно остаётся. Выигрыш двух указателей здесь не в порядке роста, а в более прямом проходе после сортировки.

> [!NOTE]
> Для этой конкретной задачи вариант через бинарный поиск действительно часто проще придумать на олимпиаде.
>
> В решении из контеста логика очень механическая: отсортировали массив, зафиксировали правый элемент, вызвали `upper_bound` или `bisect_right` на префиксе и сразу получили число подходящих левых элементов.
>
> В решении двумя указателями код после того, как идея найдена, тоже короткий. Но здесь есть дополнительное наблюдение: если пара `(left, right)` подходит, то можно сразу прибавить целый блок из `right - left` пар. Именно этот шаг не всегда приходит в голову сразу.
>
> Поэтому сравнение здесь такое: бинарный поиск часто проще именно придумать, а два указателя дают более красивый линейный проход после сортировки. Полезно уметь оба варианта и выбирать тот, который быстрее и надёжнее вспоминается в контесте.

## 19. Чем отличаются три самых частых сюжета

> [!IMPORTANT]
> На этом месте особенно важно не смешать в голове три похожих, но всё-таки разных сюжета. В коде они могут выглядеть родственниками, а по идее решают разные задачи.

**Фиксированное скользящее окно**:

- длина окна равна ровно $k$;
- левая и правая границы двигаются почти синхронно;
- основная цель — быстро обработать все окна одной длины.

**Два указателя на одном массиве**:

- длина окна меняется;
- правая граница расширяет окно, левая граница чинит его;
- чаще всего это задачи на максимальную или минимальную длину, а также на количество хороших подотрезков.

**Два указателя после сортировки**:

- окна может не быть вообще;
- указатели используются для сравнения значений и отбрасывания бесполезных вариантов;
- это типично для задач на пары, пересечения, слияния, жадное сопоставление.

Именно это различие стоит научиться распознавать первым делом, ещё до написания кода.

## 20. Когда эти методы не работают

И, наконец, нужна оборотная сторона всей лекции: по каким признакам надо вовремя остановиться и не тянуть сюда окно или два указателя.

Есть несколько типичных причин, по которым фиксированное окно или два указателя применять нельзя.

1. Условие не монотонно при расширении или сужении окна.
2. Для удаления левого элемента нельзя быстро обновить нужную характеристику.
3. В задаче на суммы есть отрицательные числа, и из-за этого при движении границ исчезает удобная монотонность.
4. На самом деле нужно отвечать на много независимых запросов к разным отрезкам, а не идти по массиву одним проходом.

В таких ситуациях часто нужны уже другие идеи:

- префиксные суммы;
- словари и структуры поиска;
- стеки, очереди на минимум, деревья отрезков и так далее.

> [!WARNING]
> То есть важно не только уметь применять два указателя, но и вовремя понимать, когда это не их задача.
