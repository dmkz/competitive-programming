## 14. Два указателя после сортировки

До сих пор оба указателя двигались по одному и тому же массиву слева направо и обслуживали текущий отрезок. После сортировки картина меняется.

Теперь два указателя могут вообще не описывать окно. Их задача другая: смотреть на упорядоченные данные так, чтобы каждым движением отбрасывать сразу целую группу бесполезных вариантов.

После сортировки появляются ещё две очень частые схемы:

- один указатель идёт с начала массива, а второй — с конца;
- по одному указателю идёт в каждом из двух отсортированных массивов.

В этих задачах уже нет “окна” в привычном смысле. Указатели нужны не для поддержки текущего подотрезка, а для того, чтобы отбрасывать заведомо бесполезные варианты целыми группами.

## 15. Пример: существует ли пара с суммой $x$

Самый классический пример такого типа — поиск пары с заданной суммой.

Пусть дан массив целых чисел и число $x$. Нужно понять, существует ли пара элементов с суммой ровно $x$.

После сортировки массива берём два указателя:

- `left` стоит в начале;
- `right` стоит в конце.

Если

$$
a_{left} + a_{right} < x,
$$

то левый элемент слишком мал, и его нужно увеличить, то есть сдвинуть `left` вправо.

Если

$$
a_{left} + a_{right} > x,
$$

то правый элемент слишком велик, и его нужно уменьшить, то есть сдвинуть `right` влево.

Если же сумма равна $x$, ответ найден.

<details>
<summary><strong>Полная программа на C++</strong></summary>

```cpp
#include <algorithm>
#include <iostream>
#include <vector>
using namespace std;

int main() {
    int n;
    long long x;
    cin >> n >> x;
    vector<long long> a(n);
    for (int i = 0; i < n; i++) {
        cin >> a[i];
    }

    sort(a.begin(), a.end());

    int left = 0;
    int right = n - 1;
    bool ok = false;
    while (left < right) {
        long long sum = a[left] + a[right];
        if (sum == x) {
            ok = true;
            break;
        }
        if (sum < x) {
            // Левый элемент слишком мал, увеличиваем сумму.
            left++;
        } else {
            // Правый элемент слишком велик, уменьшаем сумму.
            right--;
        }
    }

    cout << (ok ? "YES" : "NO") << '\n';
}
```

</details>

<details>
<summary><strong>Полная программа на Python3</strong></summary>


Если сортировка уже дана, то сам проход двумя указателями занимает $O(n)$. Если сортировку нужно сделать самим, то полная сложность равна $O(n \log(n))$.

## 16. Пример: пересечение двух массивов

Следующий стандартный сюжет — два указателя уже не на одном массиве, а на двух отсортированных массивах сразу.

Пусть даны два массива. Нужно вывести все общие элементы с учётом кратности.

После сортировки обоих массивов берём два указателя:

- `i` идёт по первому массиву;
- `j` идёт по второму массиву.

Если:

- $a_i = b_j$, то этот элемент входит в пересечение, и оба указателя сдвигаются;
- $a_i < b_j$, то нужно двигать `i`;
- $a_i > b_j$, то нужно двигать `j`.

> [!TIP]
> В C++ этот же шаблон уже есть в стандартной библиотеке: это `std::set_intersection()` для двух отсортированных диапазонов.

<details>
<summary><strong>Полная программа на C++</strong></summary>

```cpp
#include <algorithm>
#include <iostream>
#include <vector>
using namespace std;

int main() {
    int n, m;
    cin >> n >> m;
    vector<int> a(n), b(m);
    for (int i = 0; i < n; i++) {
        cin >> a[i];
    }
    for (int i = 0; i < m; i++) {
        cin >> b[i];
    }

    sort(a.begin(), a.end());
    sort(b.begin(), b.end());

    vector<int> common;
    int i = 0;
    int j = 0;
    // Сравниваем текущие элементы и двигаем тот указатель, где значение меньше.
    while (i < n && j < m) {
        if (a[i] == b[j]) {
            common.push_back(a[i]);
            i++;
            j++;
        } else if (a[i] < b[j]) {
            i++;
        } else {
            j++;
        }
    }

    cout << common.size() << '\n';
    for (int k = 0; k < (int)common.size(); k++) {
        if (k > 0) {
            cout << ' ';
        }
        cout << common[k];
    }
    cout << '\n';
}
```

</details>

<details>
<summary><strong>Полная программа на Python3</strong></summary>

## 17. Пример: слияние двух отсортированных массивов

Есть ещё один очень частый сюжет с двумя указателями на двух массивах: нужно не найти общие элементы, а слить оба отсортированных массива в один общий отсортированный массив.

Пусть даны два массива, каждый уже отсортирован в порядке неубывания.

Заведём два указателя:

- `i` идёт по первому массиву;
- `j` идёт по второму массиву.

Пока оба указателя стоят на допустимых позициях, сравниваем текущие элементы:

- если $a_i \le b_j$, берём элемент из первого массива и двигаем `i`;
- иначе берём элемент из второго массива и двигаем `j`.

Когда один из массивов закончился, остаётся просто дописать хвост второго.

Это и есть операция `merge` из сортировки слиянием. В C++ для неё есть и готовая функция `std::merge`, но руками этот шаблон тоже надо уметь писать.

<details>
<summary><strong>Полная программа на C++</strong></summary>

```cpp
#include <iostream>
#include <vector>
using namespace std;

int main() {
    int n, m;
    cin >> n >> m;
    vector<int> a(n), b(m);
    for (int i = 0; i < n; i++) {
        cin >> a[i];
    }
    for (int j = 0; j < m; j++) {
        cin >> b[j];
    }

    vector<int> merged;
    merged.reserve(n + m);

    int i = 0;
    int j = 0;
    while (i < n && j < m) {
        if (a[i] <= b[j]) {
            // Берём меньший текущий элемент из первого массива.
            merged.push_back(a[i]);
            i++;
        } else {
            // Берём меньший текущий элемент из второго массива.
            merged.push_back(b[j]);
            j++;
        }
    }

    while (i < n) {
        merged.push_back(a[i]);
        i++;
    }
    while (j < m) {
        merged.push_back(b[j]);
        j++;
    }

    for (int k = 0; k < (int)merged.size(); k++) {
        if (k > 0) {
            cout << ' ';
        }
        cout << merged[k];
    }
    cout << '\n';
}
```

</details>

<details>
<summary><strong>Полная программа на Python3</strong></summary>


## 18. Ещё один пример: количество пар с ограниченной суммой

Этот сюжет уже встречался в [задаче E из первого тематического контеста](../../01/contest/README.md). В самой задаче нужно посчитать количество пар $(i, j)$, где $i < j$ и $a_i + a_j \le s$.

Если нужен строгий вариант $a_i + a_j < s$, то идея остаётся той же самой: меняется только знак в проверке.

В тематическом контесте эта задача решалась через бинарный поиск: сортируем массив, фиксируем правый элемент пары и бинарным поиском считаем, сколько левых элементов подходят на префиксе.

Но после сортировки здесь можно пройти и двумя указателями.

Поставим:

- `left` в начало массива;
- `right` в конец массива.

Дальше смотрим на сумму $a_{left} + a_{right}$.

Если

$$
a_{left} + a_{right} \le s,
$$

то подходит не только пара `(left, right)`, но и вообще все пары `(left, k)`, где `left < k \le right`.

Причина простая: массив отсортирован, значит для любого такого `k` выполнено $a_k \le a_{right}$, а тогда

$$
a_{left} + a_k \le a_{left} + a_{right} \le s.
$$

Значит, можно сразу прибавить к ответу `right - left` пар и сдвинуть `left` вправо.

Если же

$$
a_{left} + a_{right} > s,
$$

то правый элемент слишком велик, и нужно уменьшать сумму, то есть сдвигать `right` влево.

<details>
<summary><strong>Полная программа на C++</strong></summary>

```cpp
#include <algorithm>
#include <iostream>
#include <vector>
using namespace std;

void solve() {
    int n;
    long long s;
    cin >> n >> s;
    vector<long long> a(n);
    for (int i = 0; i < n; i++) {
        cin >> a[i];
    }

    sort(a.begin(), a.end());

    int left = 0;
    int right = n - 1;
    long long answer = 0;
    while (left < right) {
        long long sum = a[left] + a[right];
        if (sum <= s) {
            // Для фиксированного left подходят все вторые индексы от left + 1 до right.
            answer += right - left;
            left++;
        } else {
            // Правый элемент слишком велик, уменьшаем сумму.
            right--;
        }
    }

    cout << answer << '\n';
}

int main() {
    int tests;
    cin >> tests;
    while (tests-- > 0) {
        solve();
    }
}
```

</details>

<details>
<summary><strong>Полная программа на Python3</strong></summary>

```python
def solve():
    n, s = map(int, input().split())
    a = list(map(int, input().split()))

    a.sort()

    left = 0
    right = n - 1
    answer = 0
    while left < right:
        current_sum = a[left] + a[right]
        if current_sum <= s:
            # Для фиксированного left подходят все вторые индексы от left + 1 до right.
            answer += right - left
            left += 1
        else:
            # Правый элемент слишком велик, уменьшаем сумму.
            right -= 1

for _ in range(tests):
    solve()
```

</details>

Файлы примера: [C++](examples/example-03.cpp), [Python3](examples/example-03.py).

После сортировки проход занимает $O(n)$, поэтому полная сложность решения равна $O(n \log(n))$.

Заметьте, что у решения из контестного разбора общая асимптотика такая же: сортировка всё равно остаётся. Выигрыш двух указателей здесь не в порядке роста, а в более прямом проходе после сортировки.

> [!NOTE]
> Для этой конкретной задачи вариант через бинарный поиск действительно часто проще придумать на олимпиаде.
>
> В решении из контеста логика очень механическая: отсортировали массив, зафиксировали правый элемент, вызвали `upper_bound` или `bisect_right` на префиксе и сразу получили число подходящих левых элементов.
>
> В решении двумя указателями код после того, как идея найдена, тоже короткий. Но здесь есть дополнительное наблюдение: если пара `(left, right)` подходит, то можно сразу прибавить целый блок из `right - left` пар. Именно этот шаг не всегда приходит в голову сразу.
>
> Поэтому сравнение здесь такое: бинарный поиск часто проще именно придумать, а два указателя дают более красивый линейный проход после сортировки. Полезно уметь оба варианта и выбирать тот, который быстрее и надёжнее вспоминается в контесте.

## 19. Чем отличаются три самых частых сюжета

> [!IMPORTANT]
> На этом месте особенно важно не смешать в голове три похожих, но всё-таки разных сюжета. В коде они могут выглядеть родственниками, а по идее решают разные задачи.

**Фиксированное скользящее окно**:

- длина окна равна ровно $k$;
- левая и правая границы двигаются почти синхронно;
- основная цель — быстро обработать все окна одной длины.

**Два указателя на одном массиве**:

- длина окна меняется;
- правая граница расширяет окно, левая граница чинит его;
- чаще всего это задачи на максимальную или минимальную длину, а также на количество хороших подотрезков.

**Два указателя после сортировки**:

- окна может не быть вообще;
- указатели используются для сравнения значений и отбрасывания бесполезных вариантов;
- это типично для задач на пары, пересечения, слияния, жадное сопоставление.

Именно это различие стоит научиться распознавать первым делом, ещё до написания кода.

## 20. Когда эти методы не работают

И, наконец, нужна оборотная сторона всей лекции: по каким признакам надо вовремя остановиться и не тянуть сюда окно или два указателя.

Есть несколько типичных причин, по которым фиксированное окно или два указателя применять нельзя.

1. Условие не монотонно при расширении или сужении окна.
2. Для удаления левого элемента нельзя быстро обновить нужную характеристику.
3. В задаче на суммы есть отрицательные числа, и из-за этого при движении границ исчезает удобная монотонность.
4. На самом деле нужно отвечать на много независимых запросов к разным отрезкам, а не идти по массиву одним проходом.

В таких ситуациях часто нужны уже другие идеи:

- префиксные суммы;
- словари и структуры поиска;
- стеки, очереди на минимум, деревья отрезков и так далее.

> [!WARNING]
> То есть важно не только уметь применять два указателя, но и вовремя понимать, когда это не их задача.

## 21. Чек-лист распознавания

> [!TIP]
> Возьмите себе привычку сразу после чтения задачи проверять монотонность ответа или монотонность данных.
>
> Если ответ монотонен по какому-то параметру, то часто можно смело применять бинарный поиск по ответу, а внутри проверки использовать скользящее окно. Если после сортировки или уже в исходных данных указатели можно двигать только в одну сторону, то очень часто подходит метод двух указателей.
>
> Это ещё не решение, но такой первый вопрос обычно очень быстро выводит на правильную технику.
